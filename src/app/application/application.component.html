Initial Application

Open your cmd and type:

ng new where-to-go -is

The -is defines that the project has an inline-style. If you don't do this, we would get an extra CSS-file for every single component. We don't want that for this project.

Next, open your editor and open the project folder. After that run the initial application. In most editors you even can open a terminal directly within the editor. If the terminal returns an error message like Command not found you have to restart it in order for the process to recognise the newly installed commands.

ng serve --open

The browser automatically opens the start page of the initial Angular app:

Angular Modules

Angular modules help structuring the application in functional blocks. Each application has at least one root module, called AppModule. Technically modules are classes like all other Angular blocks. The @NgModule() decorator marks the class as module.

Decorators define metadata which Angular uses at runtime to initialise the application and to expand the functionality of the class. The most important properties from NgModule are as follows:

  • declarations: All components, directives and pipes that belong to this module
  • exports: Specifies all components, directives and pipes which should be available for other Angular modules
  • imports: Contains all modules on which this module depends
  • providers: Creators of services, which should be globally accessible in all parts of the app
  • bootstrap: The main appliction view, called root component, which houses all other views

The main component is the starting point of the application. As you will learn in the chapter Components this is the AppComponent.

You can use different module types such as the main module, routing-module and service-module. A routing-module contains configuration for the router. You will learn about the router in the chapter Router. A service-module provides utilities like data access and messaging for the entire application as you will learn in the chapter Service.

Make some changes

Open app.component.ts and change the title of the app.

export class AppComponent {
    title = 'Where To Go';
}

In the app.component.html is a binding to title. This binding brings the new title into the application. 

<h1>Welcome to {{ title }}!</h1>

Import UI-Framework

As this is a web application showing you, how to build an Angular app, there is no need to do manual CSS styling. Probably you already know how to style with CSS and the sole focus here should be Angular. That's why we are going to use the Semantic UI CSS-Framework for a nice look and feel out-of-the-box.

You can install the UI-Framework with the command:

npm i semantic-ui-css

Tip: Its possible to open up another terminal and to have multiple running command shells to avoid powering off and on again the Angular web application.

Add it to the angular.json file:

"styles": [
    "node_modules/semantic-ui-css/semantic.css",
    "src/styles.css"
]

CSS and JS framework files with min in the file name are, as usual, minified and compressed versions for production use, in contrary to the normal files for development and debugging purposes.

Configuration changes in the angular.json file are not hot reloaded! Therefore in order for the CSS to take effect, you have to hold (Ctrl + C) and restart your app with ng serve.

Now you can test the new style. Adjust the app.component.html like this and see if it looks different:

<div style="text-align:center">
  <h1 class="ui header">
    Welcome to {{ title }}!
  </h1>
</div>

If you have any problems with installing the Semantic UI CSS, you can download the plain files from their website (under Getting Started >Simpler Setup) and paste the CSS it into your project folder.

Be aware that most UI frameworks additionally include a JavaScript file for complex widgets. These can be included too in angular.json under "scripts": []. Against this approach speaks however, that it goes against the nature of Angular as Angular should be solely responsible for rendering and manipulation of the DOM. Interferences triggered by the included framework are possible and may end up in some unwanted results.
Furthermore it may be the case that classical JavaScript frameworks tag the DOM once, to enable their functions. This normally happens after the content loaded event was fired. But this event is only emitted once in the life time of a Single-Page-Application (after the almost empty index.html was downloaded and parsed) as reloads do normally not occur. Instead the DOM is dynamically build and changed by Angular.
